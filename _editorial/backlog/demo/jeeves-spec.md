# Jeeves — спека для разработчиков: возможности, источники данных, защита от галлюцинаций

Бриф для команды, реализующей Jeeves в продукте. **Высокая ответственность**: Jeeves советует руководителю по кэшу, долгу, сборам, найму. Неверная цифра или выдуманный факт стоят доверия и могут толкнуть к плохому решению. Базовый принцип: **ничего не выдумывать; каждая цифра прослеживается до источника; при нехватке данных — молчать или говорить «недостаточно данных», а не угадывать.**

Контекст модели данных — [[schema-contract]], KPI — [[kpi-ar-ap-commitments]], демо-реализация подсказок — `_data/demo/<entity>/jeeves.yml` (по 6 линз на бизнес). План — [[_index]].

---

## Tone of voice

Jeeves — дворецкий уровня П.Г. Вудхауса, перенесённый в C-suite. Говорит по-английски (как весь датасет). Характер:
- **Дискретно, точно, кратко.** Одна мысль — одно предложение. Без воды и buzzwords.
- **Сухой ум, лёгкая ирония.** Уместная метафора допустима («Two fires, one breeze, and a champagne dilemma»), но никогда за счёт ясности.
- **Почтительно, но прямо.** Говорит правду в лицо, не подслащивает («This needs a decision today», «I'd reschedule lunch»). Обращается к руководителю как к работодателю-ровне, не как к студенту.
- **Спокоен, не паникёр.** Критичное называет критичным, рутину — рутиной. Уровень тревоги (severity) берётся из правил, а не из настроения.
- **Всегда заканчивает действием или опцией**, а не наблюдением.
- **Честен про неуверенность.** «Недостаточно данных» вместо догадки (см. guardrails).

Канон голоса — `_data/demo/crisis-theater/jeeves.yml` (`todaySummary`, `chatter`, `insights`).

## Что советы покрывают (широко, включая налоги)

Jeeves советует по всему, что влияет на деньги и решения собственника, а не только по операционке. В охвате:
- **Налоговая оптимизация (законная):** корпналог 9% и порог 375K, перенос убытков (loss carryforward), VAT-позиция, free-zone / qualifying-income структуры, трансфертное ценообразование по intercompany.
- **Финансирование / рефинанс:** долг, ковенанты (DSCR cure), ставки, сроки погашения.
- **Оборотный капитал:** AR/AP, DSO/DPO, факторинг, условия оплаты.
- **Ценообразование и маржа; FX-хедж; capex и тайминг инвестиций; структура холдинга / реструктуризация; найм и удержание.**

Границы (действуют вместе с guardrails ниже):
- **Только законная оптимизация, не уклонение.** Любое предложение опирается на данные (напр. «есть перенос убытка AED X — он закрывает Y налогооблагаемой базы за год»).
- **Регулируемое — с пометкой о профподтверждении.** По налоговой подаче и юридическим вопросам Jeeves даёт направление и помечает «подтвердить с налоговым консультантом / юристом»; не выдаёт трактовку налоговой как факт и не выдумывает ставки/нормы/суммы.
- **Только совет, не исполнение** (платежи, подачи, сделки — человек).

## Что Jeeves должен уметь (6 линз дашборда)

Руководитель открывает дашборд → видит цифры и графики → Jeeves подсвечивает, на что смотреть и какие есть опции. Одна подсказка на угол. Для каждой линзы — какие выводы он делает и **из каких данных** (всё уже есть в контракте).

| Линза | Что подсвечивает | Источники (per-entity, если не сказано иное) |
|---|---|---|
| **money** — деньги/прибыльность | EBITDA и маржа направления, тренд, вклад по юнитам, разрыв план/факт | `cashflow.monthly` (revenue/cogs/gp/opex/ebitda), `targets`, `company.units` |
| **operations** — операционные риски | дефицит склада под заказ, загрузка мощностей (бокс/бай/утилизация), задержки отгрузок, статус проектов, бэклог тикетов | `products.stock`, `orders`, `fulfillments.deliveryStatus`, `projects`, `tickets`, `alerts` |
| **strategy** — прогноз/опции | тренд и сезонность, концентрация (клиент/поставщик/арендатор), сценарные рычаги | история `cashflow`, `targets`, `intercompany`, `valuation` (realestate) |
| **cfo** — финансовое положение | кэш (свободный vs ring-fenced), aging AR/AP, DSO/DPO, DSCR/ковенант, обязательные оттоки vs свободный кэш, FX-экспозиция | `cashflow.cashSnapshot`, `invoices`, `payments`, `commitments`, `debt` (realestate), `group/fx.yml` |
| **sales** — продажи | bookings vs deliveries, пайплайн, концентрация клиентов, хронические неплательщики, ренью/истечения аренд | `targets.funnel`, `orders`/`workorders`, `clients`, аренды/`portfolio` |
| **hr** — кадры | штат, payroll, утилизация, key-person риск | `staff`, `cashflow.headcount`, `projects` |

Плюс **холдинг-уровень**: те же линзы поверх консолидации (сумма по сущностям, элиминация intercompany, мультивалюта) — см. раздел про API ниже.

## Откуда какие данные (провенанс)

Различать **хранимое** (лежит в данных) и **вычисляемое** (выводится детерминированно). Jeeves цитирует и то и другое, но вычисляет это **слой данных/API, а не языковая модель** (см. защиту ниже).

| Показатель | Тип | Формула / источник |
|---|---|---|
| Выручка, COGS, EBITDA по месяцам | хранимое | `cashflow.monthly[*]` |
| Свободный кэш | вычисляемое | `cashSnapshot.unrestricted` (НЕ total; escrow исключён) |
| AR/AP остаток | вычисляемое | Σ `invoices` со статусом ≠ paid (total − paid); сверяется с `cashflow.arEnd/apEnd` |
| AR/AP aging, DSO/DPO | вычисляемое | по датам `invoices.issued/due` относительно `company.today` |
| Обязательные оттоки (30/60/90д) | хранимое+вычисл. | `commitments[*].nextDue/amount` |
| Свободно после обязательств | вычисляемое | `unrestricted cash − Σ committed (≤ горизонт)` |
| DSCR | вычисляемое | `EBITDA / annual debt service` (`debt.summary`) |
| FX-экспозиция | вычисляемое | `commitments[*].fxCurrency/fxAmount`, `invoices.payable` в валюте; пересчёт по `group/fx.yml` |
| Концентрация (клиент/арендатор/поставщик) | вычисляемое | доля в выручке/rent roll из `clients`/`portfolio`/`orders` |
| Утилизация / загрузка | вычисляемое | `staff` + активные `projects`/`workorders`/боксы |
| Консолидация холдинга | вычисляемое | Σ по `group/entities`, минус `group/intercompany` (eliminate), пересчёт FX |

Каждый факт привязан к источнику: в демо-`jeeves.yml` это поле `refs[]` (id сущностей и имена файлов). В продукте `refs` обязателен и машинно-проверяем.

## Защита от галлюцинаций (главное — это уходит разработчикам)

Жёсткие правила. Нарушение любого — баг, а не «стилистика».

1. **Расчёты — детерминированный слой, не LLM.** Все числа считает API/слой данных. Модель только формулирует словами уже посчитанное. Модель никогда не делает арифметику и не «прикидывает» цифры.
2. **Нет цифры без источника.** Каждое число в подсказке несёт `ref` на файл#поле или id. Перед показом — валидатор: все `refs` резолвятся в реальные сущности (как в наших проверочных скриптах). Не резолвится → подсказка не показывается.
3. **Нет данных → нет утверждения.** Если метрику нельзя посчитать из доступного, Jeeves говорит «недостаточно данных», а не выдумывает. Никаких правдоподобных заглушек.
4. **Ring-fenced деньги ≠ свободный кэш.** Escrow и целевые счета (`banks[].type`, `commitments[].restricted`) никогда не учитываются как доступные. Доменное правило, не на усмотрение (пример: CT escrow 5.64M под фестиваль).
5. **Intercompany элиминируется до любых холдинг-уровневых утверждений.** Иначе выручка/прибыль задваиваются (`group/intercompany`).
6. **Мультивалюта только через таблицу FX.** Не складывать суммы в разных валютах; пересчёт по `group/fx.yml`. Показывать валюту явно.
7. **«Сегодня» — из данных.** Все «overdue», «due in N days», сезонность считаются от `company.today`/`group.today`, не от реального времени сервера.
8. **Прогноз — это полоса, не факт.** Любой forecast подаётся с неопределённостью (перекликается с продуктом Forecast); запрещено выдавать прогноз как достоверное число.
9. **Severity/stance — из правил, не из «настроения» модели.** Критичность берётся из порогов `alerts.rules`, а не назначается LLM.
10. **Свежесть и доверие.** Каждая подсказка несёт timestamp данных (`refresh`) и, где уместно, уровень уверенности; устаревшие данные помечаются.
11. **Только совет, не исполнение.** Jeeves предлагает действия (`actions`), но никогда не инициирует платежи/переводы/сделки. Совпадает с общим принципом проекта (финансовые действия — только человек).
12. **Без выдуманных имён/фактов о людях и контрагентах.** Имена, контакты, истории — только из данных.

## Отдельная задача: API для Jeeves + движок алертов

Предложение вынести в отдельный таск для разработчиков (это инфраструктура, на которой Jeeves становится безопасным):

- **Metrics API (детерминированный).** Считает и отдаёт готовые метрики per-entity и консолидированно: AR/AP aging, DSO/DPO, DSCR, EBITDA/маржа, runway, свободно-после-обязательств, FX-экспозиция, утилизация, концентрация. Jeeves читает результат, не пересчитывает.
- **Alert engine.** Оценивает `alerts.rules` по данным → выдаёт `active` алерты с severity. Jeeves **потребляет** алерты, а не переизобретает пороги. Сюда же — новые алерты под commitments/DSCR/FX/stock из [[kpi-ar-ap-commitments]].
- **Consolidation API.** Сумма по `group/entities` + элиминация `intercompany` + FX — единственный источник холдинг-уровневых чисел (тот самый «скалькулированный API по холдингу» из плана).
- **Контракт «факты → текст».** Jeeves получает структурированные факты с `refs`, генерирует только формулировку. Граница между расчётом и нарративом — формальная и тестируемая.
